1. Executive Summary

Ember represents an order-of-magnitude architectural leap over traditional interpreted LED controllers on cheap microcontrollers. Evolved from WebFastLed, it delivers stable, lag-free 60 FPS rendering for complex patterns by running a stack-based virtual machine directly inside an isolated FreeRTOS thread.

It is important to note the landscape: if you seek the absolute gold standard in scriptable LED systems, we highly recommend checking out ElectroMage's Pixelblaze. While ElectroMage's hardware costs more, it is a highly proven, rock-solid ecosystem capable of driving thousands of pixels with custom expansion boards and an exceptionally mature compiler. It is the ultimate premium option with the absolute least friction.

Ember is brand new! It is a lightweight, open-source DIY compiler still early in its maturity. But for developers and makers looking to push cheap $2 ESP32-C3 modules to their absolute limits, Ember offers an exceptionally elegant, single-chip captive portal sandbox that does far more than raw FastLED with FFT.

Made Possible by the Claude Code Hackathon: This project was accelerated into reality during the State of Oregon | Claude Code Hackathon (sponsored by Anthropic and Vibes DIY at the PSU Business Accelerator). To prepare for the event, I built my recursive command-center suite, Agent Arsenal, beforehand. Once I arrived and received the hackathon key, I dropped it in, opened Claude Code directly in my downloaded GitHub folder (without my credentials, on a freshly wiped laptop), copied prompts over from my arsenals, and gave the agents total freedom to run around with zero permissions or approvals. While I fully expected a mountain of AI slop, the massive power of this setup allowed me to completely break through the final interpreter and FreeRTOS concurrency bottlenecks—transforming a raw, unstable firmware prototype into a highly polished, production-grade 60 FPS platform!

2. Core Architectural Shifts (Compiler & VM)

                
The Bottleneck: Legacy Parser Evaluation

                

                    In traditional script-based LED systems, executing a pattern requires parsing the expression
                    character-by-character on every single pixel of every single frame. If you have 60 LEDs at
                    30 FPS, the processor scans, parses, and evaluates your formulas 1,800 times a second!
                

                

                    This is exacerbated by heap allocations for temporary token strings, std::map variables, and
                    parameter lists—leading to severe memory fragmentation and immediate thread blockages.
                

                
The Solution: Compact Stack Bytecode

                

                    Ember features a robust single-pass Pratt-like compiler built
                    directly into the firmware. When a pattern is edited:
                

                
    
                    
  1. It compiles the JS-like formula once into a compact flat array of bytecode
                        instructions.
    2. 
                    
  2. Numbers are extracted into a de-duplicated float constant pool.
    3. 
                    
  3. Variables are resolved to static uint8_t register indices, eliminating string
                        lookups.
    4. 
                    
  4. The runtime is a hyper-streamlined stack-based virtual machine executing lightweight opcodes on
                        pre-allocated buffers.

3. Task Isolation & Concurrency (RTOS)

                

                    The ESP32-C3 is a single-core RISC-V processor. Standard multitasking libraries cannot run on
                    separate cores. Ember solves this by building a dedicated **FreeRTOS task
                    schedule** that guarantees flawless output:
                

                
    
                    
  • RenderTask (Priority 2): Runs locked to a stable 60Hz loop using
                        hardware-timed delays.
    • 
                    
  • AsyncTCP Worker (Priority 3): Set higher than the renderer.
                        Network updates or compiler requests preempt the renderer for microseconds, keeping the web
                        dashboard snappy without causing a single frame to stutter.
    • 
                    
  • Atomic Lock-Free Exchange: When a pattern is compiled, it is pushed to a
                        std::atomic<Program*> pointer swap. The renderer captures it atomically at
                        the top of the next frame, avoiding data corruption.

4. Real-Time FFT Spectral Analyzer

                

                    The audio reactivity module handles floating-point math seamlessly:
                

                
    
                    
  • Reads digital INMP441 MEMS mics via I2S, or analog MAX4466
                        mics via ADC1.
    • 
                    
  • Computes a 256-point Hamming-windowed Forward FFT via arduinoFFT.
    • 
                    
  • Compresses frequency bins into 8 logarithmic bands.
    • 
                    
  • Provides built-in functions: vu(), peak(), pitch(),
                        beat(), and band(n).
